# 핫딜 레이더 (Hotdeal Radar) — 프로토타입

해외직구 핫딜을 **원천 소스에서 먼저 잡아 우리 딜보드에 선제 발행**하는 파이프라인의 최소 동작 프로토타입.
설계 근거: `../DEAL_FILTER_SPEC.md`, 데이터 분석: `../DEAL_DATA_ANALYSIS.md`.

```
hotdeal_radar/
├─ data/
│  └─ watchlist.csv      # 감시 대상 185개 리테일러 (실측+딜공 병합, robots 점검 완료)
├─ scraper/
│  ├─ scheduler.py       # 상주 스케줄러 (티어별 주기 폴링 + FLASH 즉시 알림)
│  ├─ notify.py          # 텔레그램 알림
│  ├─ pricing.py         # 환율 정규화 · 관부가세 · 국내 최저가 (H2 하드필터)
│  ├─ store.py           # SQLite (중복 방지 + 가격 이력 축적)
│  ├─ watchlist.py       # 워치리스트 레지스트리 (티어별 폴링 주기·세일페이지 경로·수집 전략)
│  ├─ parsers.py         # 사이트별 전용 파서 (브랜드·평점·리뷰수 정확 추출)
│  ├─ renderer.py        # JS 렌더 사이트용 헤드리스 브라우저 (Playwright)
│  ├─ filter_engine.py   # 하드필터·스코어링·긴급도 (스펙 2~4단계)
│  ├─ sources.py         # 수집 어댑터 (워치리스트 크롤 / DoA RSS / Slickdeals API 슬롯)
│  ├─ run.py             # 파이프라인 엔트리 → web/deals.json 생성
│  └─ requirements.txt
└─ web/
   ├─ index.html         # 딜보드 (deals.json을 읽어 렌더)
   └─ deals.json         # 발행 큐 (run.py가 생성)
```

## 감시 대상 (watchlist.csv)

**슬릭딜을 거치지 않고 실제 판매 사이트를 직접 감시**하는 것이 이 프로젝트의 핵심이다.
목록은 두 소스를 병합해 만들었다:

1. **4년치 카톡 실측** — 실제로 올린 링크에서 추출한 리테일러 도메인 136개 (`posts` = 실제 게시 횟수)
2. **딜공 몰 디렉토리** 97개

→ 중복 제거 후 **185개**. 각 사이트는 robots.txt를 사전 점검해 수집 전략이 정해져 있다:

| 전략 | 사이트 수 | 처리 |
|---|---|---|
| `direct_crawl` | 114 | robots 허용 → 세일페이지 직접 크롤 |
| `affiliate_feed` | 65 | 봇탐지/차단(HTTP 403 등) → 제휴 네트워크 피드로 우회 |
| `excluded` | 6 | robots 전면 차단 (전부 국내몰이라 대상 외) |

티어는 **실제 게시 빈도** 기반이고 폴링 주기를 결정한다:

| 티어 | 기준 | 주기 | 예시 |
|---|---|---|---|
| T1 | 100건+ | 15분 | Amazon(788), Woot(206), eBay(160) |
| T2 | 15~99건 | 30분 | Macy's(84), Nordstrom Rack(42), Nike(40), Adidas(40), Columbia(27), Zappos(23), The Outnet(23), Jomashop(20), REI(18) |
| T3 | 2~14건 | 2시간 | 45개 |
| T4 | 0~1건 | 6시간 | 127개 (딜공 목록 위주) |

```bash
python watchlist.py     # 목록 요약과 T1/T2 감시 URL 확인
```

## 상시 감시 실행 (스케줄러)

```bash
cd scraper
python scheduler.py                    # 상주 루프 (T1 15분 / T2 30분 / T3 2h / T4 6h)
python scheduler.py --once             # 1사이클만 (테스트)
python scheduler.py --once --tier T1   # 특정 티어만
```

### 발행 정책

- 🔴 **FLASH** (85점+ 또는 slickdeals frontpage) → **시각 무관 즉시 푸시**. 좋은 딜일수록 수명이 짧다.
- 🟠 **HOT** / 🟡 **STEADY** → 정기 슬롯(**08~10 / 13~16 / 20~23시**)에만 발송.
  슬롯 밖에서 발견되면 대기열에 담아뒀다가 다음 슬롯에 묶어서 보낸다.
- 한 번에 5건을 넘으면 개별 알림 대신 **다이제스트 한 통**으로 묶는다(알림 피로 방지).

### 텔레그램 알림 설정 (5분)

1. 텔레그램에서 `@BotFather` → `/newbot` → 봇 이름 입력 → **토큰** 받기
2. 만든 봇과 대화 시작(아무 메시지나 전송)
3. `https://api.telegram.org/bot<토큰>/getUpdates` 접속 → `chat.id` 확인
4. 환경변수 설정 후 실행:

```bash
set TELEGRAM_BOT_TOKEN=123456:ABC...
set TELEGRAM_CHAT_ID=123456789
python scheduler.py
```

토큰이 없으면 **콘솔 출력으로 자동 폴백**하므로 설정 전에도 그대로 동작한다.

### 실질가 판정 — 환율·관부가세·국내 최저가 (pricing.py)

**해외직구는 표시가격이 최종가가 아니다.** 싸 보여도 관부가세를 더하면 국내가와 역전되는 일이 흔하다.
스펙 H2가 잡으려는 게 이 지점이고, `pricing.py`가 이를 실제로 작동시킨다.

1. **환율 정규화** — eBay는 한국 접속 시 KRW, 나머지는 USD를 준다. 통화가 섞이면 비교가 어긋나므로
   모두 원화로 환산한다. 환율은 무료 API 2곳(open.er-api, frankfurter)에서 가져와 6시간 캐시한다.
2. **관부가세** — 미국발은 면세한도 **USD 200**(그 외 국가 150). 넘으면 *초과분이 아니라 전체 금액*에
   관세+부가세가 붙는다. 직구에서 가장 자주 헷갈리는 지점이라 계산에 반영했다.
3. **국내 최저가** — 네이버 쇼핑 검색 API로 조회. 이상치(짝퉁·부품·중고)에 걸리지 않도록
   최저가 하나가 아니라 하위 절반의 중앙값을 쓴다.

```bash
set NAVER_CLIENT_ID=...        # 네이버 개발자센터, 무료 · 일 25,000회
set NAVER_CLIENT_SECRET=...
```

키가 없으면 국내가 조회만 건너뛰고 나머지는 정상 동작한다. API 쿼터를 아끼려 할인율 상위 25건에만 조회한다.

**실측 검증** (환율 1,486.9원 기준):

| 사례 | 표시가 | 실질 원화가 | 국내 최저가 | 절감 | 판정 |
|---|---|---|---|---|---|
| Kate Spade Kara Loafer | $34.99 | 52,028원 | 197,110원 | **+73.6%** | ✅ 80점 HOT |
| 같은 상품 고가 시나리오 | $145 | 215,608원 | 197,110원 | −9.4% | ❌ H2 탈락 |
| New Balance 990v6 | $210 | **388,138원** | 216,400원 | −79.4% | ❌ H2 탈락 |

> 세 번째가 이 필터의 존재 이유다. $260→$210이면 19% 할인처럼 보이지만,
> 면세한도 $200을 넘겨 전액 과세되면서 **국내에서 사는 것보다 79% 비싸진다.**
> 영어 검색어(`Kate Spade New York Kara Loafer`)로 한국어 상품(`케이트스페이드 카라 로퍼`)이
> 정확히 매칭되는 것도 확인했다.

### 중복 방지 & 가격 이력 (store.py)

같은 딜이 15분마다 다시 잡히므로 중복 방지가 없으면 알림 폭탄이 된다.
SQLite에 알림 이력을 남기고, 다시 알리는 경우는 둘 뿐이다:

- 가격이 직전 알림가보다 **5% 이상 더** 내려갔을 때
- 마지막 알림 후 **14일**이 지났을 때

동시에 매 스캔의 가격을 스냅샷으로 쌓는다. **스펙 3-B(가격 히스토리 25점)는 외부 서비스가 아니라
우리가 직접 쌓아야 작동**하기 때문이다. 관측 3회 이상 **그리고** 기간 3일 이상이 모이면
해당 상품은 콜드스타트 보정이 꺼지고 정상 채점으로 자동 전환된다.

> 기간 조건이 왜 필요한가: 횟수만 보면 15분 간격으로 3번 찍은 이력도 '유효'로 판정돼
> 모든 상품이 역대 최저가가 되고 점수가 부풀려진다(테스트 중 실제로 발생해 조건을 추가함).

**DB 위치**: 클라우드 동기화 폴더의 SQLite는 잠금 문제로 `disk I/O error`가 나거나 손상될 수 있어
기본값이 `%LOCALAPPDATA%\hotdeal_radar\radar.db`(동기화 폴더 밖)다. `RADAR_DB`로 변경 가능.

## 헤드리스 렌더링 (Playwright)

Amazon·eBay·Woot·Nike는 상품을 JS로 그린다(실측: Amazon 골드박스 556KB HTML에 가격 9개뿐).
`renderer.py`가 실제 브라우저로 열어 JS 실행 후 DOM을 가져온다.

```bash
pip install playwright
playwright install chromium
playwright install-deps chromium     # 리눅스: 시스템 라이브러리
```

Playwright가 없으면 자동으로 정적 fetch로 폴백하므로 파이프라인은 계속 동작한다.
렌더링은 페이지당 수 초가 들어, **전용 파서가 등록된 사이트에만** 적용된다(파서 없이 렌더링해봐야 추출 불가).

### 범용 파서 (generic_parser.py)

전용 파서가 없는 사이트는 다섯 전략을 순서대로 시도하고 **가장 많이 뽑히는 것을 채택**한다.

| 전략 | 대상 | 확인된 사이트 |
|---|---|---|
| `jsonld` | schema.org Product 구조화 데이터 | — |
| `semantic` | `data-test="productPrice0"` 형태 | SSENSE (60건) |
| `searchspring` | Searchspring 검색 플랫폼(`ss__result`) | Shoebacca (24건) |
| `class` | `product-name` / `product-oldprice` 관례 | Eddie Bauer, END |
| `card` | 상품 카드 블록 + 가격 두 개 | 폴백 |

`searchspring`처럼 **상용 플랫폼을 지원하면 그걸 쓰는 여러 사이트가 한꺼번에 열린다.**
새 사이트가 안 잡히면 마크업을 보고 해당 전략을 확장하는 게 사이트별 파서보다 효율적이다.

정적 fetch로 0건이면 **자동으로 헤드리스 렌더링을 재시도**한다.
상당수 사이트가 요청마다 다른 HTML을 주기 때문이다(실측: SSENSE는 같은 URL 연속 호출에서
상품 120개 → 0개로 바뀌었다).

### 사이트별 파서 추가하기

`parsers.py`에 함수를 쓰고 `REGISTRY`에 도메인을 등록하면 끝이다.

```python
def mysite(html, site):
    return [{"url":..., "title":..., "brand":..., "image":...,
             "price_current":..., "price_list":...}, ...]

REGISTRY = {..., "mysite.com": mysite}
```

`image`는 상품 썸네일 URL이다. 딜보드 카드에 표시되며, 없으면 "이미지 없음"으로 나온다.
공개 사이트에서는 이미지 유무가 클릭률에 크게 영향을 주므로 가능하면 넣는다.

브랜드를 정확히 넣는 게 핵심이다 — 브랜드가 없으면 스펙 3-C 카테고리 점수가 `etc`(5점)로 떨어져 발행 기준(50점)을 못 넘는다.

### Amazon 특이사항 (중요)

Amazon 딜 그리드는 **가격을 지연 하이드레이션**한다. 같은 URL을 같은 설정으로 렌더해도
가격 JSON(`priceToPay`)이 붙을 때와 안 붙을 때가 갈린다(실측 확인). 그래서:

- `renderer.CONTENT_SIGNAL`에 도메인별 "쓸모 있는 렌더" 신호를 두고, 신호가 없으면 **자동 재시도**한다(기본 2회).
- 스크롤 대기(`RADAR_SCROLL_WAIT_MS`, 기본 500ms)가 수집량을 좌우한다. Amazon은 700ms 이상을 권장.
- `wait_until="networkidle"`은 쓰지 말 것 — Amazon은 백그라운드 폴링이 계속돼 영원히 안 끝난다.

```bash
RADAR_SCROLL_WAIT_MS=700 python run.py
```

장기적으로 Amazon은 **Product Advertising API**(Associates 계정 필요)가 정답이다.
Slickdeals과 같은 패턴 — 공식 API가 있으면 스크래핑보다 안정적이고 약관상 안전하다.

### 디스크 관리 (중요)

헤드리스 브라우저는 페이지마다 임시 캐시를 만든다. 여유가 없으면 렌더링만 실패하는 게 아니라
**DB 쓰기까지 터져 사이클 전체가 죽는다**(실측: `OSError: [Errno 28] No space left on device`로
203건을 수집해놓고 저장 단계에서 중단).

그래서 세 겹으로 막아뒀다.

1. 렌더링 전에 **여유 공간을 확인**하고, 부족하면 임시 파일을 자동 정리한다
2. 그래도 부족하면 **렌더링만 건너뛰고** 정적 수집은 계속한다
3. 저장 단계에서 디스크 오류가 나면 **모은 딜을 잃지 않고** 정리 후 안내만 남긴다

기본 하한은 **1.5GB**다. 스케줄러 시작 시 현재 여유가 표시되며, 빠듯하면 경고가 붙는다.

```
setx RADAR_MIN_FREE_MB 3000      # 하한을 3GB로 올리려면
```

여유가 계속 부족하면 윈도우 `디스크 정리`로 임시 파일을 비우거나,
`%LOCALAPPDATA%\ms-playwright` 의 오래된 브라우저 버전을 지우면 된다.

### 현재 한계 (다음 작업)

1. **Nike는 아직 0건** — 렌더링 인프라는 붙었으나 전용 파서가 없다. `parsers.py`에 추가하면 즉시 살아난다.
   Columbia는 세일 경로가 robots 차단, Jomashop은 HTTP 에러 → 둘 다 제휴 피드 경로 검토 대상.
2. **콜드스타트** — 가격 이력 DB가 비어 있어 B항목(25점)을 아무도 못 받는다. 현재는 A·C·D(75점)로 채점 후 100점 척도로 정규화해 보정 중이며, 이력이 쌓이면 자동으로 정상 채점으로 전환된다.
3. **지역 통화** — eBay는 접속 지역을 감지해 한국에서는 KRW·한국어 제목을 준다. 파서가 통화를 보존하고 `krw_effective`에 넣어 H2(국내가 역전) 판정에 쓰지만, USD 기준 사이트와 섞이므로 환율 정규화가 필요하다.

## 동작 원리 (4단계)

1. **수집**: 소스 어댑터가 URL·가격을 `Deal` 객체로 변환.
2. **하드필터** (H1~H5): 배송불가/국내가역전/무명셀러/품절/리퍼 즉시 컷.
3. **스코어링** (100점): 할인깊이 40 + 가격이력 25 + 카테고리 20 + 보너스 15.
4. **긴급도**: 85+ 또는 slickdeals frontpage=🔴FLASH, 70+=🟠HOT, 50+=🟡STEADY, 50 미만 미발행.

발행 큐는 점수순으로 `deals.json`에 저장되고 딜보드가 실시간 렌더한다.

## 다음 단계 (프로덕션화)

- `run.py`를 스케줄러(cron/APScheduler)로 주기 실행: 핵심 브랜드 15~30분, 나머지 2~4시간.
- **가격 이력 DB**: 매 스캔 스냅샷을 SQLite에 적재해야 스펙 B항목(역대/1년/90일 최저가)이 작동. 90일 누적 전까지는 A·C·D 위주 스코어링.
- **링크-팔로우 파서**: 실제 게시의 97%가 맨 링크였으므로, 소스에서 URL을 얻으면 그 페이지를 2차 fetch/렌더해 브랜드·가격·할인·쿠폰을 추출 (`DEAL_DATA_ANALYSIS.md` 참고).
- 발행 채널 연동: 딜보드 외에 카톡/텔레그램/웹푸시로 FLASH 즉시 알림.
